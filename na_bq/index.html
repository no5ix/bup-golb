What is ETL?

ETL stands for Extract, Transform, Load.
It’s a common data integration process used to move and prepare data for analytics and machine learning:

  • Extract: Pull data from source systems (e.g., databases, APIs, flat files).
  • Transform: Clean, filter, enrich, or aggregate data into the desired format.
  • Load: Store the transformed data into a target system, like a data warehouse or data lake.

In the context of AWS, ETL workflows are often orchestrated using tools like Glue and Step Functions.

Amazon SageMaker

A fully managed machine learning platform that:

  • Trains and tunes ML models.
  • Hosts models for real-time or batch inference.
  • Handles model versioning, scaling, and A/B testing.

In projects, we use SageMaker to deploy trained models behind APIs or for batch inference jobs.

AWS Glue

A serverless ETL service that:

  • Crawls and catalogs data from S3 or databases.
  • Runs PySpark or Python jobs to transform data at scale.
  • Integrates with Data Catalog and Athena for querying.

It’s used to automate and scale data preprocessing pipelines.

Amazon S3 (Simple Storage Service)

A durable object storage service that:

  • Stores raw, processed, or intermediate data.
  • Hosts model artifacts, training datasets, and logs.
  • Acts as a data lake foundation in most pipelines.

It’s the main storage layer for both data and ML artifacts.

AWS Step Functions

A serverless orchestration service that:

  • Coordinates tasks (e.g., Glue jobs, Lambda, SageMaker, etc.) into stateful workflows.
  • Adds retry, timeout, and error handling logic to ETL and ML pipelines.
  • Makes pipelines maintainable and scalable.

We use it to build and visualize complex, multi-step data or ML workflows.

Amazon API Gateway

A fully managed service for creating, securing, and scaling APIs:

  • Exposes REST or WebSocket endpoints to external clients.
  • Integrates with Lambda, Step Functions, or SageMaker endpoints.
  • Handles rate limiting, logging, and authentication.

It’s commonly used to serve model predictions or expose backend services securely.

A1: Latency, ordering, and fault tolerance. We used compacted topics for quote updates and partitioned topics by instrument ID for order executions. Consumers had exactly-once semantics via transactional reads and offset commits.

Q2: How did you handle backpressure in Kafka consumers?
A2: We used bounded queues with flow control and applied manual offset commits only after downstream processing succeeded. In extreme load, we activated circuit breakers to shed non-essential processing.

6. Built secure REST APIs with Spring Security, OAuth2, and JWT

Q1: How did you design your API’s security model for trading-related operations?
A1: Role-based access control via OAuth2 scopes was enforced using method-level @PreAuthorize. JWTs carried user claims and expiration info. For order placement, we required 2FA tokens validated against a Redis-backed OTP service.

Q2: How did you handle JWT expiration and refresh in real-time systems?
A2: We used short-lived access tokens with long-lived refresh tokens. Refresh was triggered in the background using Axios interceptors in the frontend, and token revocation lists were stored in Redis to prevent misuse.

7. Optimized PostgreSQL and Redis for fast access to trading and market data

Q1: What specific query or indexing strategies did you use for PostgreSQL?
A1: We used partial indexes on active portfolios, GIN indexes for full-text search on tickers, and materialized views for frequently queried aggregates. We also analyzed slow queries weekly and tuned work_mem and effective_cache_size.

Q2: How was Redis used in your system, and what issues did you face?
A2: Redis was used to store hot order book snapshots and user session tokens. We had to manage eviction policies carefully (using LRU) and isolate critical keys with different Redis DBs to avoid performance interference.

8. Automated CI/CD for real-time trading services using Jenkins, Docker, Kubernetes

Q1: What did your CI/CD pipeline look like, and how did you ensure safety?
A1: We had multi-stage Jenkins pipelines with static checks, unit tests, integration tests, canary deployments, and health checks. Rollbacks were handled via Helm charts versioning and blue/green deployments.

Q2: How did you handle configuration and secrets in your pipeline?
A2: We used Vault for secret management, injected via environment variables during deployment. Configs were templated with Helm and stored in GitOps-style repositories to keep infra reproducible.

9. Monitored trading system health with Prometheus and Grafana

Q1: What key metrics did you monitor for trading system reliability?
A1: Latency (P50, P95, P99), order rejection rates, Kafka lag, Redis hit ratio, and API error rates. Alerts were configured using Prometheus Alertmanager with Slack integrations.

Q2: How did you debug latency spikes during market open or earnings hours?
A2: We used distributed tracing (OpenTelemetry) to track slow spans, correlated spikes with CPU/memory metrics, and used Grafana’s heatmaps to identify hot services. We also performed GC tuning and thread pool analysis.
